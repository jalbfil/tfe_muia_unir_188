\chapter{Capturas del prototipo}
\label{cap:anexog}

Este anexo presenta de forma ilustrada la interfaz de usuario desarrollada para interactuar con el sistema de tres capas. A través de las imágenes seleccionadas, se documentan las distintas pantallas con las que trabaja el despachador de emergencias. Con este soporte visual se pretende facilitar la comprensión práctica del flujo operativo completo de la herramienta.

\section*{G.1. Finalidad del anexo}

Este anexo incorpora la documentación visual del prototipo software v0.1.0, obtenida sobre una ejecución local estable del backend, la interfaz Streamlit y Ollama local. El objetivo es mostrar evidencias visuales del flujo descrito en los Capítulos~\ref{cap:8}, \ref{cap:9} y \ref{cap:10}, abordando desde la entrada manual y por voz hasta la explicación local y las probabilidades P1--P4. Gracias a esta recopilación, se ilustra la interacción en tiempo real del operador con la aplicación. De esta forma, el lector puede apreciar la disposición y diseño de los componentes del panel gráfico.

Las capturas no sustituyen a los artefactos reproducibles ni a las métricas. Su función es facilitar la defensa del TFM mostrando que la arquitectura no se queda en una descripción abstracta, sino que existe una interfaz capaz de ejecutar el flujo completo de entrada textual, priorización RuleFit-lite, explicación local y registro de feedback. Esta demostración visual sirve para constatar la viabilidad de la integración tecnológica propuesta. Asimismo, aporta un valor didáctico relevante de cara a la exposición oral del trabajo fin de máster.

\section*{G.2. Criterios aplicados}

Las imagenes incorporadas cumplen los siguientes criterios:

\begin{itemize}
    \item proceder de una ejecucion local del prototipo v0.1.0;
    \item capturarse sobre una ejecucion estable con datos simulados sinteticos para
    evitar exposicion de datos reales del Registro 112 CyL;
    \item no contener datos personales, telefonos, direcciones completas ni informacion
    sensible de personas reales;
    \item usar incidentes sinteticos o casos anonimizados;
    \item mostrar claramente la prioridad recomendada, la distribucion de confianza y
    la explicacion generada cuando la vista de resultados lo permite;
    \item permitir verificar que Capa 2 decide la prioridad y que Capa 3 solo la
    explica;
    \item estar alineadas con el smoke test P1--P4 documentado en
    \texttt{artifacts/reports/prototype\_v0.1.0\_smoke\_test.md} y con la validacion
    reproducible Capa 1--Capa 2--Capa 3 documentada en
    \texttt{artifacts/reports/capa1\_capa2\_e2e\_v0.1.0.md}.
\end{itemize}

\section*{G.5. Relacion con la evidencia experimental}

Las capturas deben interpretarse como evidencia de usabilidad y ejecucion del prototipo,
no como evaluacion cuantitativa. La evaluacion principal sigue siendo la del
Capitulo~\ref{cap:9}: comparativa baseline experto, RuleFit-lite e \texttt{imodels},
recall P1, test temporal, control anti-\textit{leakage}, validacion interna y fidelidad
explicativa. La captura de variables de Capa 1 debe servir especialmente para conectar
la interfaz con la validacion \texttt{capa1\_capa2\_e2e\_v0.1.0}, donde el flujo
integrado supera 5/5 escenarios curados tras el saneamiento de negaciones.

Por tanto, este anexo completa la memoria con soporte visual del MVP y conecta la implementación ejecutable con la evaluación experimental y la trazabilidad humana. Con esta aportación gráfica se cierra el ciclo de documentación del software de apoyo a la decisión. Se ofrece con ello una perspectiva de ingeniería práctica que enriquece el valor del trabajo académico.
